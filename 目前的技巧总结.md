# 目前的技巧总结

> 更新时间：2026-05-16
> 目的：总结当前项目已经验证过、后续还能继续复用的有效技巧与经验

---

## 1. 方案层面的技巧

### 1.1 先把问题改写成“可交付问题”

这次最重要的判断不是“理论上能不能做全量子 4D-Var”，而是：

- 两天内什么最可能做出来；
- 什么既合规又能讲清楚；
- 什么能真正产出 `result.csv`。

当前验证下来，最有效的思路是：

- **经典主流程保底**
- **量子模块只接管一个规模可控的关键子问题**

这比一开始直接追求全量子方案更现实。

### 1.2 先做强经典基线，再谈量子增益

如果经典基线本身很弱，量子模块即使提升一点，也很难说明问题。

当前有效策略是：

1. 先把 `LETKF` 跑通；
2. 先把参数调到一个足够强的水平；
3. 再让量子模块去接这部分参数选择任务。

这样最后可以形成更有说服力的对比：

- 不是“量子版 vs 弱经典版”
- 而是“量子版 vs 经典强基线”

### 1.3 不要夸大量子模块的作用

当前最成立的说法是：

- 量子模块负责**参数调优**
- `LETKF` 负责**主状态估计**

这是真实、稳定、可复现的描述。

不建议写成：

- “量子模块直接求解了完整 4D-Var”
- “当前系统已经实现了完整量子数据同化”

## 2. 数据与评估层面的技巧

### 2.1 官方没给评分公式时，先用常规误差指标统一口径

当前最有效的做法是：

- 主指标：`RMSE`
- 辅指标：`MAE`

因为赛题的本意已经很明确：

- 误差越低越好

所以在没有更明确公开公式前，统一用 `RMSE / MAE` 做对比是合理的。

### 2.2 只对有真值的重叠样本打分

当前数据和预测文件可能在时间步长度上不完全一致，因此必须统一口径：

- 只对**同时存在真值和预测值的重叠行**计算误差

这个技巧很重要，否则：

- 容易误算；
- 容易把不同长度文件硬拉齐；
- 会导致前后结果不可比较。

### 2.3 先和朴素基线比，再和强基线比

当前最清晰的三层对比结构是：

1. 朴素基线：直接用 `observed_value`
2. 经典强基线：人工优化后的 `LETKF`
3. 量子调参版：量子选参后的 `LETKF`

这样对内、对外都容易讲清楚：

- 数据同化本身是否有效；
- 经典优化是否有效；
- 量子模块是否继续带来收益。

## 3. LETKF 调参层面的技巧

### 3.1 不要只盯着 `infl`

一开始容易只围绕 `infl` 小范围调参，但这次真正带来明显变化的是：

- `seed`
- `ens_size`
- `loc_radius`

特别是：

- `seed` 对结果有实际影响；
- `ens_size` 增大后结果确实变好；
- `loc_radius` 增大后效果反而持续改善。

### 3.2 这份数据上，弱局部化不一定更好

一个很关键的实测结论是：

- `loc_radius` 从较小值增大到接近全局时，误差持续下降

这意味着当前数据场景下：

- 强局部化未必占优；
- 更接近全局更新反而更有效。

这点后面写说明文档时可以作为一个经验观察。

### 3.3 `dt` 和 `forcing` 不要乱动

当前反复试下来，稳定结论是：

- `dt = 0.05`
- `forcing = 8.0`

这两个参数和题目设置是一致的，而且没有显示出明显需要偏离的证据。

所以真正值得优化的是：

- 同化器超参数

而不是盲目偏离问题本身的动力学设定。

## 4. 量子模块设计层面的技巧

### 4.1 量子模块最适合先做“离散候选选择”

当前最稳的切入点不是直接量子化 40 维状态，而是：

- 先把一小组候选参数组合离散化；
- 再让量子模块去做候选概率分布建模；
- 最终选择最优候选。

这样做的好处是：

- qubits 需求小；
- 逻辑清楚；
- 更容易快速跑通。

### 4.2 候选数控制在 16 或 32 最舒服

因为：

- `16` 个候选对应 `4` qubits；
- `32` 个候选对应 `5` qubits。

这个范围有几个好处：

- 远低于比赛的 `30` qubits 限制；
- 候选空间不至于太小；
- 运行时间还能接受；
- 方便用 `2^n` 方式直接编码。

### 4.3 量子模块要嵌到“能解释”的位置

当前量子模块之所以成立，不只是因为用了量子电路，而是因为它的位置合理：

- 它直接参与了参数选择；
- 参数选择又直接影响 `LETKF` 最终结果；
- 所以量子模块与最终误差之间是有路径关联的。

这个点比“量子电路长什么样”更重要。

### 4.4 量子结果不必强行比经典网格搜索更低

一个很重要的表达技巧是：

- 当前量子模块的目标不是证明“量子一定比经典穷举更强”；
- 而是证明“量子模块可以在受控候选空间中完成有效选择，并落到真实可用参数上”。

在当前阶段，只要能做到：

- 量子选择与网格内经典最优一致；
- 或量子选择结果在测试集上确实可用；

这条技术路线就已经成立。

## 5. 远程开发与执行层面的技巧

### 5.1 对远程交互终端，不要依赖一次性大段命令

这次反复验证下来，远程壳最稳的做法是：

- 小步执行；
- 单步落文件；
- 单步检查产物；
- 能后台就后台；
- 跑完再读日志。

不要指望一次性把很长的 here-doc 和多段逻辑一起灌进去就稳定成功。

### 5.2 上传脚本最稳的是 base64

对远程环境而言，最可靠的上传方式之一是：

1. 本地生成脚本；
2. 本地转 `base64`；
3. 远程解码；
4. 再执行。

这个方法适合：

- 脚本较长；
- here-doc 易被壳吞掉；
- 需要避免字符转义混乱。

### 5.3 长任务尽量后台跑

像：

- 大网格搜索
- 量子调参
- 测试集完整推理

这类任务更适合：

- `nohup ... > run.log 2>&1 &`

然后再轮询：

- 进程状态
- 日志内容
- 结果文件是否落盘

### 5.4 先读产物，不要迷信终端回显

很多时候远程命令已经跑完了，但终端回显没有完整回来。

更靠谱的方法是：

- 直接去看输出文件；
- 直接去看日志；
- 直接去看 `summary.json` 或 `.csv` 是否存在。

## 6. 代码与生态兼容层面的技巧

### 6.1 当前 Qiskit 路线优先使用 `StatevectorEstimator`

在已有环境里，`StatevectorEstimator` 是已经验证过的可用路线。

当前已知经验：

- `BackendEstimatorV2` 兼容性更麻烦；
- `StatevectorEstimator + scipy.optimize` 更稳。

所以如果后面还要继续做量子局部优化，优先继续沿这条路。

### 6.2 在比赛文档里要承认“当前量子模块是混合式”

这不是缺点，反而是更合理的工程现实。

正确表达应是：

- 经典外循环 + 量子内核
- 混合量子-经典优化

而不是把它包装成完全脱离经典优化器的纯量子系统。

## 7. 文档与答辩层面的技巧

### 7.1 叙事顺序要固定

当前最顺的叙事顺序是：

1. 赛题本质是数据同化，不是单纯量子电路炫技；
2. 经典方法负责稳定主流程；
3. 量子方法嵌入关键子问题；
4. 最终误差确实下降；
5. 方案可运行、可复现、可提交。

### 7.2 不要把“真实提升很小”说成“革命性突破”

当前量子调参版相对经典强基线的提升是：

- 小幅但真实

这种表述最稳：

- 不夸大；
- 不容易被质疑；
- 更符合比赛型项目的真实水平。

### 7.3 一定要把技巧沉淀成文档

这次已经证明，像下面这些经验如果不及时记下来，后面很容易重复踩坑：

- 远程终端怎么跑最稳；
- 哪些参数最值得调；
- 量子模块放在哪个位置最合理；
- 当前最好的对比口径是什么。

因此重要阶段结束后，务必把关键信息写进：

- `jindu.md`
- 对比说明
- 算法说明文档
- 技巧总结

## 8. 目前最值得保留的核心结论

如果只保留最关键的几条经验，当前最值得记住的是：

1. **比赛本质是受约束的数据同化问题，不是单纯量子算法展示。**
2. **两天时间内最稳路线是 LETKF 主骨架 + 量子参数调优。**
3. **必须先把经典基线做强，再谈量子增强。**
4. **当前数据上，大 `loc_radius` 比强局部化更有效。**
5. **量子模块最适合先做离散候选选择，不要一上来做全量子状态估计。**
6. **远程执行时，小步、后台、日志、产物检查，比长命令回显更可靠。**
7. **当前最适合对外汇报的结果是：`test_1 RMSE = 0.12616446776255305`。**
